너는 Draft 프로젝트의 배포 전 Phase 1 체크리스트를 수행하는 에이전트야.
작업 디렉토리: C:\project\Draft\main

## 절대 규칙
- 절대 git push 하지 마. 커밋만 해.
- 절대 main 브랜치에서 작업하지 마. deploy/phase1-fixes 브랜치에서만 작업해.
- 절대 package.json, package-lock.json을 수정하지 마.
  **단, Rate Limit Redis 전환을 위해 `@upstash/ratelimit` 패키지만 설치 허용**
  (`npm install @upstash/ratelimit` 명령어는 반드시 사용하고, 다른 패키지는 절대 추가 금지)
- 절대 DB 마이그레이션 파일을 만들거나 수정하지 마.
- 절대 기존 컴포넌트의 디자인/스타일을 변경하지 마.
- 절대 .env 파일을 읽거나 수정하지 마.

## 작업 흐름
1. deploy/phase1-fixes 브랜치 생성 후 체크아웃
2. 각 항목마다:
   a. 관련 파일을 먼저 읽고 현재 상태 파악
   b. 최소한의 변경만 수행
   c. npx next build 실행하여 빌드 확인
   d. 빌드 성공 → 해당 파일만 git add + commit (항목별 개별 커밋)
   e. 빌드 실패 → git checkout -- . 으로 되돌리고 실패 사유 기록, 다음 항목으로
3. 모든 항목 완료 후 결과 기록

## 작업 목록 (이 순서대로만 수행, 한 작업 끝난 후 반드시 next build 실행)

### 1. Rate Limit → Upstash Redis 전환 (최우선)
- npm install @upstash/ratelimit 실행
- src/lib/rate-limit/redis-rate-limiter.ts 새로 생성 (기존 api-rate-limiter.ts는 유지)
- @upstash/ratelimit + Redis.fromEnv() 사용
- 기본 정책: slidingWindow(10, "60 s")
- **우선 적용 대상**:
  • 모든 AI 엔드포인트 (onboarding/chat, onboarding/summarize, onboarding/parse, help/chat, ai-chat/complete 등)
  • view count API 2개 (opportunities/[id]/view, profile/[id]/view)
- 각 route.ts 상단에서 `await ratelimit.limit(userId || ip)` 호출 추가
- 적용 후 next build 성공 확인 필수

### 2. next.config.ts 이미지 hostname 수정 (15분)
- images.remotePatterns에서 hostname: '**' 완전 제거
- 허용 도메인만 남겨: Supabase storage 도메인 (*.supabase.co), GitHub avatar (avatars.githubusercontent.com, github.com), Google avatar (lh3.googleusercontent.com)
- 기존에 이미 있는 도메인들은 유지하면서 '**'만 삭제

### 3. CSP 헤더 추가 (30분)
- middleware.ts에 response.headers.set('Content-Security-Policy', "...") 추가
- 안전한 기본 정책 사용:
  default-src 'self';
  script-src 'self' 'unsafe-inline' 'unsafe-eval' https://www.googletagmanager.com;
  style-src 'self' 'unsafe-inline';
  img-src 'self' https: data:;
  font-src 'self' https://fonts.gstatic.com;
  connect-src 'self' https://*.supabase.co wss://*.supabase.co https://api.anthropic.com https://generativelanguage.googleapis.com;
  frame-ancestors 'none';
- 기존 보안 헤더들(X-Content-Type-Options 등) 바로 옆에 추가

### 4. 온보딩 리다이렉트 변경 (5분)
- 온보딩 완료 후 `/dashboard`로 보내는 부분을 `/explore`로 변경
- 관련 파일: components/Onboarding.tsx, app/onboarding/page.tsx, middleware.ts 등에서 onboarding 완료 후 redirect 경로 검색
- router.push('/dashboard') 또는 redirect('/dashboard') 패턴 찾아서 변경

### 5. 빈 페이지 네비게이션 숨김 (30분)
- dashboard, calendar, network, documents 4개 페이지가 아직 빈 스켈레톤
- TopNavbar 또는 Sidebar에서 이 4개 페이지로의 링크/탭을 숨기거나 제거
- 페이지 파일 자체는 삭제하지 말고, 네비게이션에서만 숨겨
- 만약 직접 URL 접근 시에는 "준비 중입니다" 안내를 표시해도 좋음

### 6. robots.ts 추가 (30분)
- app/robots.ts 파일 생성
- /admin, /api, /onboarding, /dev, /project/build, /project/ideate 경로 Disallow
- /, /explore, /p/* 경로 Allow
- sitemap URL 포함

### 7. sitemap.ts 추가 (30분)
- app/sitemap.ts 파일 생성
- 정적 페이지: /, /explore, /login, /waitlist
- 동적 페이지: /p/[id] — Supabase에서 공개 opportunities 조회하여 URL 생성
- Supabase import: import { createClient } from '@supabase/supabase-js' 사용
- 환경변수: process.env.NEXT_PUBLIC_SUPABASE_URL, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY

### 8. NEXT_PUBLIC_APP_URL 폴백 도메인 통일 (15분)
- 코드베이스에서 'dailydraft.kr', 'dailydraft.vercel.app' 하드코딩된 부분을 찾아서
- 모두 process.env.NEXT_PUBLIC_APP_URL || 'https://dailydraft.me' 로 통일

### 9. (dashboard/) 라우트 그룹 수정 (15분)
- app/(dashboard/)/ 디렉토리의 파일들을 app/(dashboard)/ 로 이동
- app/(dashboard/)/admin/page.tsx → app/(dashboard)/admin/page.tsx
- app/(dashboard/)/admin/opportunities/page.tsx → app/(dashboard)/admin/opportunities/page.tsx
- app/(dashboard/)/admin/users/page.tsx → app/(dashboard)/admin/users/page.tsx
- 주의: app/(dashboard)/admin/ 에 이미 invite-codes, error-logs가 있으므로 충돌 없는지 확인
- 이동 후 app/(dashboard/)/ 빈 디렉토리 삭제
